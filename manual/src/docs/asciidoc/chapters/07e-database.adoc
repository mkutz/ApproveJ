[id=database]
= Database -- Catch Changes to Queries and Data

The database extension provides two complementary approaches to approval testing with databases:

* **Approve the SQL** your code sends — catch unintended changes to queries, just like the <<http,HTTP extension>> catches changes to HTTP requests.
No test database setup required.
* **Approve the query results** — verify the actual state in the database after your code runs.

[cols="1,1"]
|===
|Approve SQL |Approve query results

|Catches changes in _what your code sends_
|Catches changes in _what ends up in the database_

|No test data or database state needed
|Requires a database with predictable state

|Uses `RecordingDataSource` + `SqlPrintFormat`
|Uses `DatabaseSnapshot` + `QueryResultPrintFormat`
|===


[id=database_dependencies]
== Dependencies

The `database` module has no external dependencies beyond the JDK's `javax.sql` API.
You need to provide a JDBC driver for your database (e.g., H2 for in-memory testing, or the driver for your production database).

== Approve SQL

When your application talks to a database, a code change can silently alter the SQL it sends — a different `WHERE` clause, a missing `JOIN`, a changed column list.
The `RecordingDataSource` intercepts and records all SQL statements your code executes, so you can approve them.

[id=database_approve_sql_wiring]
=== Wiring the RecordingDataSource in Spring Boot

Since `RecordingDataSource` implements `DataSource`, it can be passed anywhere a `DataSource` is expected.
With **plain JDBC**, simply wrap your `DataSource` before passing it to the code under test, as shown in the <<database_approve_sql,basic example>> above.

In a **Spring Boot** test, the `DataSource` is managed by the container and injected into your services automatically.
The following steps show how to set this up.


[id=database_approve_sql_wiring_step1]
==== Step 1: Create a test class with your service

Start with a `@SpringBootTest` that injects the service you want to test and the `DataSource`:

[source,java,indent=0,role="primary"]
.Java
----
@SpringBootTest
class OrderServiceTest {

  @Autowired
  DataSource dataSource;

  @Autowired
  OrderService orderService;
}
----
[source,kotlin,indent=0,role="secondary"]
.Kotlin
----
@SpringBootTest
class OrderServiceTest(
  @Autowired val dataSource: DataSource,
  @Autowired val orderService: OrderService,
)
----

icon:check-circle[] _Checkpoint: the test class compiles and Spring can inject both beans._


[id=database_approve_sql_wiring_step2]
==== Step 2: Wrap the DataSource with RecordingDataSource

Add a `@TestConfiguration` with a `BeanPostProcessor` that replaces the `DataSource` bean with a `RecordingDataSource` wrapper.
This ensures all beans — including your services and JPA — use the recording wrapper transparently.

[source,java,indent=0,role="primary"]
.Java
----
@SpringBootTest
@Import(OrderServiceTest.RecordingConfig.class) // <1>
class OrderServiceTest {

  @TestConfiguration
  static class RecordingConfig {
    @Bean
    static BeanPostProcessor recordingDataSourceWrapper() {
      return new BeanPostProcessor() {
        @Override
        public Object postProcessAfterInitialization(Object bean, String beanName) {
          if (bean instanceof DataSource ds && !(bean instanceof RecordingDataSource)) {
            return new RecordingDataSource(ds);
          }
          return bean;
        }
      };
    }
  }

  @Autowired
  DataSource dataSource; // <2>

  @Autowired
  OrderService orderService;
}
----
<1> `@Import` activates the configuration. Without it, the inner class is ignored.
<2> The injected `DataSource` is now the `RecordingDataSource` wrapper.

[source,kotlin,indent=0,role="secondary"]
.Kotlin
----
@SpringBootTest
@Import(OrderServiceTest.RecordingConfig::class) // <1>
class OrderServiceTest(
  @Autowired val dataSource: DataSource, // <2>
  @Autowired val orderService: OrderService,
) {

  @TestConfiguration
  class RecordingConfig {
    companion object {
      @Bean @JvmStatic
      fun recordingDataSourceWrapper() = BeanPostProcessor { bean, _ ->
        if (bean is DataSource && bean !is RecordingDataSource)
          RecordingDataSource(bean) else bean
      }
    }
  }
}
----
<1> `@Import` activates the configuration. Without it, the inner class is ignored.
<2> The injected `DataSource` is now the `RecordingDataSource` wrapper.

icon:check-circle[] _Checkpoint: the context starts and `dataSource instanceof RecordingDataSource` is `true`._


[id=database_approve_sql_wiring_step3]
==== Step 3: Write the approval test

Cast the `DataSource` to `RecordingDataSource`, reset it, exercise your code, and approve the last recorded query using the <<database_sql_print_format,SQL print format>>:

[source,java,indent=0,role="primary"]
.Java
----
@Test
void place_order() {
  RecordingDataSource recordingDataSource = (RecordingDataSource) dataSource;
  recordingDataSource.resetRecordedQueries();

  orderService.placeOrder(new Order("Widget", 3));

  approve(recordingDataSource.lastRecordedQuery())
      .printedAs(sql())
      .byFile();
}
----
[source,kotlin,indent=0,role="secondary"]
.Kotlin
----
@Test
fun place_order() {
  val recordingDataSource = dataSource as RecordingDataSource
  recordingDataSource.resetRecordedQueries()

  orderService.placeOrder(Order("Widget", 3))

  approve(recordingDataSource.lastRecordedQuery())
    .printedAs(sql())
    .byFile()
}
----

icon:check-circle[] _Checkpoint: the first run creates a `-received.sql` file next to the test.
Review it, rename it to `-approved.sql`, and re-run — the test passes._

TIP: Call `resetRecordedQueries()` before exercising the code under test to ensure you only capture the SQL you care about.


[id=database_sql_print_format]
=== Pretty-Print SQL

The link:{javadoc-url}/database/org/approvej/database/SqlPrintFormat.html[`SqlPrintFormat`] formats SQL with line breaks and indentation, making it easy to review in approved files.
Files are saved with the `.sql` extension.

A query like `SELECT u.id, u.name FROM users u WHERE u.active = 1 ORDER BY u.name` becomes:

[source,sql]
----
SELECT
  u.id,
  u.name
FROM
  users u
WHERE
  u.active = 1
ORDER BY
  u.name
----

[id=database_approve_state_test_isolation]
=== Test Isolation

Approving database state only works if the data is predictable.
When multiple tests write to the same database, execution order or parallel runs can cause unexpected results.

Common strategies to keep tests isolated:

* **Separate database per test** — Use a fresh in-memory database (e.g., H2 with a unique URL per test) so each test starts with a clean slate.
* **Transaction rollback** — Run each test in a transaction and roll back after the assertion.
  Spring's `@Transactional` test support does this by default.
* **Reset between tests** — Truncate or re-seed tables in a `@BeforeEach` method.
* **Targeted queries** — Use `WHERE` clauses to query only the data created by the current test, so other rows don't affect the result.

Regardless of which strategy you use, scrub any non-deterministic column values (auto-incremented IDs, timestamps) so the approved file stays stable.
